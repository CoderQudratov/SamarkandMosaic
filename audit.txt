LOCK STRATEGY MISMATCH — CRITICAL BUG:
  build_levels.py defines lock strategy per level:
    L2-5  → 'rows'    (top + bottom rows locked)
    L6-10 → 'corners' (4 corner tiles locked)
    L15+  → 'none'

  SwapLevelLoader.detectLockedSlots() defines it by levelNum:
    levelNum <= 4  → rows  (top + bottom rows)
    levelNum <= 8  → corners
    levelNum >= 9  → none

  CONFLICT ON LEVEL 5:
    build_levels.py says L5 = 'rows' lock (top+bottom rows locked)
    level-5/level.json marks top+bottom row tiles as locked: true
    detectLockedSlots(5, 4, 4) returns: CORNERS (0, 3, 12, 15)

  The locked field in level.json is COMPLETELY IGNORED by SwapLevelLoader.
  Line 199 in SwapLevelLoader.ts:
    const lockedSlots = detectLockedSlots(levelNum, rows, cols);
  This overwrites whatever the JSON says. Level 5 will present corner pieces as
  locked (allowing top/bottom row pieces to be swapped — breaking puzzle design).

SECTION 4 — PUZZLE LOGIC REPORT
────────────────────────────────────────────────────────────────────────────────

SHUFFLE: createShuffledGrid() uses Fisher-Yates on movable slots only. ✓
  Locks are respected during shuffle. ✓
  Anti-trivial guard: recursion if solved. ✓
  [WARN] Infinite loop risk: if movable tiles ≤ 1, isGridSolved returns true
    on every call → infinite recursion → stack overflow. No guard.

SWAP DETECTION: isGoodSwap() returns true if either tile ends up within
  Manhattan distance ≤ 1 of its correct slot after the swap.
  [NOTE] "Within 1" means already-adjacent tiles are counted as good even
    if they don't actually reach their target. This is a design choice that
    may be intentional (ASMR feel — low penalty) but means the penalizer
    (bad swap) is rarely triggered for most mid-game positions.

SOLVE DETECTION: grid.every((t, i) => t === i + 1) — correct. ✓

BEST MOVE HINT (findBestMove):
  O(n²) scan — acceptable for n ≤ 36 (6×6 max). ✓
  Prefers mutual-improvement swaps over single-tile swaps. ✓
  Final fallback: returns first two movable slots unconditionally (even if no
    improvement available). This means the hint can suggest a bad swap if no
    good moves exist — which then costs the player an attempt.

isTileImproved checks distance AFTER swap, not delta-distance.
  Edge case: tile already at distance 1 from correct slot swaps to distance 1
  on the other side — counts as "improved" but player makes no net progress.
  Penalty is skipped even though puzzle state hasn't improved meaningfully.

SHUFFLE RECOVERY (loadSwapState):
  Validates grid.length === expectedLength and attemptsLeft is a number. ✓
  Does NOT validate that grid contains valid tile IDs (no range check on values).
  A corrupted save with grid=[0,0,0...] would pass validation and break render.

SECTION 5 — ECONOMY REPORT
────────────────────────────────────────────────────────────────────────────────

ECONOMY DATA CHAIN:
  build_levels.py writes manifest.json → economy block
  SwapLevelLoader reads manifest.json → LoadedLevel.economy
  SwapPuzzleBoard reads LoadedLevel ... and IGNORES economy.maxAttempts

CRITICAL BUG — maxAttempts NEVER APPLIED FROM MANIFEST:

  SwapPuzzleBoard.tsx lines 403, 607, 1012:
    const maxAtt = SWAP_MAX_ATTEMPTS[selectedLevelId] ?? 12;

  SWAP_MAX_ATTEMPTS in SwapSystem.ts:
    export const SWAP_MAX_ATTEMPTS: Record<number, number> = { 1: 12 };

  Only level 1 is in this table. ALL OTHER LEVELS get maxAttempts=12.
  The manifest.json economy.maxAttempts values (20 for L2, 26 for L3,
  34 for L4, 42 for L5, 72 for L10, 132 for L20) are LOADED but DISCARDED.

  Impact: Level 20 (6×6=36 tiles, 32 movable) gets 12 attempts instead of 132.
  At E[min_swaps] ≈ 32 × ln(32) / 2 ≈ 56 swaps, level 20 is UNSOLVABLE.

  The fix is one line: replace SWAP_MAX_ATTEMPTS lookup with data.economy.maxAttempts.

COIN ECONOMY:
  perSnap = 5 coins per correct placement
  completion base = 50 coins (from EconomySystem.calcLevelReward)
  manifest.coinReward is loaded into economy but check needed in EconomySystem
  [NOT VERIFIED] Whether EconomySystem actually uses economy.coinReward from
    the manifest or uses CONFIG.coins.completion globally.

HINT PURCHASE UI: hardcoded pack prices [75, 300, 800, 1800] coins.
  These are independent of manifest.economy.hintCosts ([25, 40, 60, 90]).
  The hintCosts in manifest describe "per-use cost when buying with coins during
  gameplay" — but that code path (spendHint) is never called from SwapPuzzleBoard.
  Instead, the board calls usePlayerStore.getState().useHint() which only deducts
  from inventory. Coin-based hints would require no inventory.
  Result: hintCosts in manifest are defined but never consumed.

STARS:
  3 stars: no mistakes + no hint used
  2 stars: mistakes but no hint
  1 star:  hint used
  This is implemented correctly via mistakesRef and hintUsed from swapStore. ✓

REWARD CHEST:
  30% chance after win. Min=30, max=100 coins. ✓
  ChestModal shows then routes to win scene. ✓

SECTION 6 — DETECTED BUGS
────────────────────────────────────────────────────────────────────────────────

[BUG-01] CRITICAL — Economy maxAttempts never applied to board
  File: src/game/board/SwapPuzzleBoard.tsx:403, 607, 1012
  File: src/game/systems/SwapSystem.ts:4
  Description: SWAP_MAX_ATTEMPTS only has level 1. All other levels get 12.
    Manifest economy.maxAttempts is loaded but discarded.
    Level 20 is unsolvable at 12 attempts (needs ~56 minimum).
  Fix: Replace:
    const maxAtt = SWAP_MAX_ATTEMPTS[selectedLevelId] ?? 12;
  With:
    const maxAtt = data.economy.maxAttempts;

[BUG-02] CRITICAL — Lock strategy mismatch for level 5
  File: src/game/loaders/SwapLevelLoader.ts:66-79
  File: scripts/build_levels.py:59-71
  Description: L5 build_levels uses 'rows' lock (top+bottom rows).
    detectLockedSlots(5, ...) returns corners (since levelNum <= 8).
    level.json for L5 has locked:true on top/bottom tiles.
    But the locked field is ignored — loader always calls detectLockedSlots().
    Result: L5 shows wrong tiles as locked. Puzzle design is corrupted.
  Fix: Read locked slots from level.json tiles[].locked instead of
    detectLockedSlots(), or fix the boundary mismatch (L5 → rows).

[BUG-03] HIGH — Levels 7 and 9 will crash at load time
  File: public/assets/levels/level-7/, public/assets/levels/level-9/
  Description: Only level.json exists in these directories (no tiles/, no
    manifest.json, no board.png). SwapLevelLoader will 404 on tile images and
    throw "missing tiles array" if the JSON schema doesn't match.
  Fix: Either build L7 and L9 via the pipeline, or remove them from the
    level registry / lock them on the level select screen.

[BUG-04] HIGH — Infinite recursion in createShuffledGrid
  File: src/game/systems/SwapSystem.ts:17-26
  Description: If movable tile count is 0 or 1, isGridSolved returns true
    immediately and the function recurses forever → stack overflow.
  Fix: Guard before recursion:
    if (movableSlots.length <= 1) return grid; // can't shuffle

[BUG-05] MEDIUM — level.json locked field completely ignored
  File: src/game/loaders/SwapLevelLoader.ts:199
  Description: All tile metadata including locked:true/false is loaded from
    level.json via validateTiles(), but the locked property is not read.
    detectLockedSlots() is called instead, ignoring the JSON data.
    This makes build_levels.py's locked flag writing pointless.
  Fix: In SwapLevelLoader.loadLevel(), derive lockedSlots from tiles:
    const lockedSlots = new Set(tiles
      .filter(t => t.locked)
      .map(t => t.correctRow * cols + t.correctCol));
    And remove the detectLockedSlots() call (or use it only as fallback).

[BUG-06] MEDIUM — handleReplay uses wrong fallback row count
  File: src/game/board/SwapPuzzleBoard.tsx:1010
  Code: const rows = data?.manifest.rows ?? 7;
  Description: No level in the game has 7 rows. This fallback would create
    a wildly wrong grid. If levelData is null on replay (shouldn't happen
    but defensive), the board initializes with 4×7=28 slots for all levels.
  Fix: Use rows ?? 4 or assert data is non-null before this point.

[BUG-07] MEDIUM — findBestMove suggests bad swap as final fallback
  File: src/game/systems/SwapSystem.ts:82-83
  Code: return [movable[0], movable[1]];
  Description: When no good move exists, hint returns any two movable slots.
    If the player accepts this hint, it executes a bad swap, costs an attempt,
    and the hint lamp shows it as a "smart" suggestion. Misleading and harmful.
  Fix: Return null when no improving swap exists, and display a "no good moves"
    message instead of suggesting a guaranteed bad swap.

[BUG-08] LOW — loadSwapState does not validate tile ID range
  File: src/game/systems/SwapSystem.ts:98-103
  Description: Saved grid values are not checked against [1..total].
    A corrupted localStorage entry with out-of-range IDs passes validation
    and will produce blank tiles (tileImageMap miss) silently.
  Fix: Add: if (raw.grid.some(v => v < 1 || v > expectedLength)) return null;

[BUG-09] LOW — audioStore uses localStorage directly, not storageService
  File: src/store/audioStore.ts:31-38
  Description: All other stores use storageService.get/set for persistence.
    audioStore calls localStorage.getItem/setItem directly. If storageService
    ever changes its storage backend or prefix, audio prefs won't migrate.
  Fix: Replace localStorage calls with storageService calls for consistency.

[BUG-10] LOW — CONFIG.levels only defines levels 1-10, game has levels 1-20
  File: src/constants/config.ts (CONFIG.levels object)
  Description: CONFIG.levels has pieces counts for L1-10 only. This is the
    OLD system's config. Levels 15 and 20 are missing. If any new code
    reads CONFIG.levels[15] it gets undefined silently.
  Fix: Extend to all 20 levels or mark as explicitly OLD-system-only.

[RED-01] ECONOMY IS BROKEN FOR MOST LEVELS
  Every level except level 1 gets 12 attempts regardless of difficulty.
  Level 10 (5×5, E[min_swaps]≈42) has 12 attempts → impossible to win cleanly.
  Level 20 (6×6, E[min_swaps]≈56) has 12 attempts → unsolvable.
  This is the single highest-priority bug. Players who reach level 3+ will
  be force-failing without understanding why.

[RED-02] DUAL SYSTEM CREATES MAINTENANCE DEBT AND CONFUSION
  Two loaders, two board components, two game stores, two swap budgets.
  The OLD system (LevelLoader, PuzzleBoard, gameStore) appears unmaintained.
  BootLoader still preloads level-1 via the OLD loader — if OLD loader
  triggers on every app start it wastes network requests.
  RECOMMEND: Audit whether OLD system is still reachable from any navigation
  path. If not, remove it. Dead code + stale stores = audit noise + future bugs.

[RED-03] LEVELS 7 AND 9 ARE BROKEN SHELLS
  They exist in public/ with no tiles and no manifest. Any player who reaches
  L7 or L9 gets a crash. There is no level-select guard preventing access.

[RED-04] LEVEL-1 IS NOT QA'D
  The only level reachable by new players is explicitly excluded from the
  automated QA audit (LEGACY_LEVELS = {1}). Level 1's 28 tiles with
  1254×1254 master are untested by the pipeline.

[RED-05] LOCK MISMATCH CORRUPTS PUZZLE DESIGN FOR LEVEL 5
  A player on L5 will see corner tiles "locked" when the ornament was designed
  for top/bottom row anchoring. This makes the puzzle visually incoherent.
